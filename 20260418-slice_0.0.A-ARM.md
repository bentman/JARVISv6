# 20260418-slice_0.0.A-ARM.md
## Slice 0.0.A — ARM64 Voice-Capable Path (CPU-EP Baseline)

**Date:** 2026-04-18
**Status:** Proposed
**Depends on:** Slice 0.0 Hardware Profile Provisioning (Verified), Slice 6 Durable Desktop Host (Verified), provisioning correction closeout 2026-04-18 (Verified)

---

## Slice 0.0.A Goal

Make JARVISv6 voice-capable on Windows ARM64 hosts using CPU-EP only.

Current repo truth after the provisioning correction:

- `backend/requirements.txt` is platform-neutral and installs cleanly on ARM64
- `config/hardware/hw_x64_base.json` correctly gates `onnxruntime`, `faster-whisper`, `kokoro`, `misaki[en]` to `arch: amd64`
- ARM64 hosts match `hw-cpu-base` and (when present) `hw-npu-present`, neither of which currently adds any Python packages
- the profiler already recommends `onnx-whisper` for Qualcomm hosts (`_recommend_stt()` in `backend/app/hardware/profiler.py`), but no concrete `onnx-whisper` STT runtime exists in `backend/app/runtimes/stt/`
- the only concrete TTS runtime is `KokoroTTSRuntime` in `backend/app/runtimes/tts/local_runtime.py`, which depends on the `kokoro` + `torch` path that is unavailable on ARM64
- STT and TTS readiness on ARM64 therefore resolve to `selected_device: unavailable` with `selected_device_ready: False` — fail-closed, correct, but not voice-capable

This slice closes that gap by introducing an additive ARM64 manifest, a concrete ONNX-backed STT runtime, a concrete ONNX-backed TTS runtime, and the minimum catalog/selector wiring needed to surface both through existing selector boundaries.

**What Slice 0.0.A is not:** an NPU acceleration slice, a runtime-architecture redesign, a replacement for the existing x64 CUDA path, or an application-surface change. Desktop shell, resident session service, turn engine, cognition, memory, artifacts, interruption, wake-word host integration, and the validated x64/CUDA readiness rail are all consumed unchanged. This is purely an additive runtime-availability slice for a second host class.

---

## Why this slice exists now

The repo can install cleanly on ARM64, detect ARM64 hardware correctly, and profile it to sensible recommendations. It cannot yet execute a voice turn on ARM64 because the runtime implementations the recommendations point to do not exist. Without this slice, ARM64 remains install-capable but not voice-capable, which contradicts the `ProjectVision.md` commitment to voice-first execution across hardware classes.

The provisioning correction closeout on 2026-04-18 established the right boundary for hardware-specific packages to enter; this slice is the first consumer of that boundary.

---

## Scope

This slice covers:

- one new hardware manifest: `config/hardware/hw_arm64_base.json`, arch-gated, additive with `onnxruntime` and `kokoro-onnx`
- one new concrete STT runtime: ONNX Whisper via `onnxruntime` CPU-EP, implementing `STTBase`
- one new concrete TTS runtime: Kokoro via `kokoro-onnx`, implementing `TTSBase`
- model catalog extensions for the new runtimes in `config/models/stt.yaml` and `config/models/tts.yaml`
- selector extensions in `backend/app/runtimes/stt/stt_runtime.py` and `backend/app/runtimes/tts/tts_runtime.py` so profiler recommendations route to the new runtimes
- evidence-token extensions in `backend/app/hardware/profiler.py::resolve_backend_evidence_tokens()` and the STT/TTS readiness derivers in `backend/app/hardware/preflight.py`
- unit coverage under the existing `backend/tests/unit/test_hardware_readiness.py` and new STT/TTS unit surfaces mirroring the existing slice-1/2 patterns
- one live acceptance test per backend family, executed on an ARM64 host

## Out of scope

This slice does not:

- introduce QNN, Qualcomm NPU, or any NPU-accelerated execution provider (that is Slice 0.0.B)
- modify `hw_x64_base.json`, `hw_gpu_nvidia_cuda.json`, `hw_gpu_amd.json`, or any already-matching manifest applicability
- modify the existing `FasterWhisperSTT` or `KokoroTTSRuntime` implementations
- modify `backend/requirements.txt` beyond confirming it remains universal
- modify Slice 6 desktop surfaces (`desktop/src-tauri/**`, `desktop/src/**`), `ResidentSessionService`, backend API routes, or any session/turn/artifact code path
- modify the canonical turn executor, personality, memory, cognition, or interruption layers
- retire `FasterWhisperSTT` or `KokoroTTSRuntime` — they remain the x64 path and must stay green
- introduce new manifest merge semantics (replacement, exclusion, priority) — additive-only is preserved
- add DirectML, WebGPU, or any other non-CPU execution provider path
- modify wake-word runtime behavior beyond confirming the existing degraded-mode path is honored when no ARM64-compiled `.ppn` is available

## Architectural Approach

### The manifest carries ARM64 runtime packages

`hw_arm64_base.json` mirrors the shape of `hw_x64_base.json`:

```json
{
  "manifest_id": "hw-arm64-base",
  "version": "0.0.1",
  "merge_behavior": "additive",
  "applies_when": { "arch": "arm64" },
  "additive_requirements": {
    "python_packages": [
      "onnxruntime>=1.20",
      "kokoro-onnx>=0.5.0"
    ]
  },
  "notes": "ARM64-only runtime packages. onnxruntime supplies the CPU-EP STT backend and the inference surface kokoro-onnx consumes. Does not apply on x64 hosts; NPU/QNN acceleration is out of scope for this manifest."
}
```

Both packages have official `win_arm64` wheels on PyPI. `kokoro-onnx` is distributed as `py3-none-any` and depends transitively on `onnxruntime`, `numpy`, `phonemizer-fork`, and `espeakng-loader` — all ARM64-compatible. No `torch`, no `ctranslate2`, no source builds.

The resolver (`backend/app/hardware/profile_resolver.py`) already supports `arch` as a condition key and already merges additively; the manifest drops in without resolver changes.

### STT runtime: ONNX Whisper over `onnxruntime` CPU-EP

A new concrete runtime under `backend/app/runtimes/stt/` implements `STTBase` the same way `FasterWhisperSTT` does:

- constructor takes `model_name` and `device` (only `cpu` supported in this slice)
- `is_available()` guards `import onnxruntime`
- `_ensure_model()` resolves the ONNX Whisper model directory via the existing `catalog.get_model_entry()` → `manager.ensure_model()` path
- `transcribe(audio_path)` runs the session with CPU-EP, returns the transcript string
- fail-closed behavior matches `FasterWhisperSTT`: missing audio, missing model, or session failure raise `RuntimeError` with an explicit reason

The selector in `stt_runtime.py` gains a second branch: `stt_recommended_runtime == "onnx-whisper"` instantiates the new runtime; `faster-whisper` keeps the existing branch. No other changes to `select_stt_runtime()`.

### TTS runtime: Kokoro via `kokoro-onnx`

A new concrete runtime under `backend/app/runtimes/tts/` implements `TTSBase` the same way `KokoroTTSRuntime` does:

- constructor takes `model_name`, `voice`, `device` (only `cpu` in this slice)
- `is_available()` guards `import kokoro_onnx`
- `_ensure_pipeline()` resolves the `kokoro-v1.0.onnx` and `voices-v1.0.bin` files via an extended catalog entry and `manager.ensure_model()` so acquisition reuses the proven HuggingFace path
- `synthesize(text, audio_path)` writes a WAV file; signature matches the existing `KokoroTTSRuntime.synthesize()` contract so `voice_service.py` is unchanged
- `HF_HUB_OFFLINE=1` is set after ensure (same pattern as the existing runtime)

The selector in `tts_runtime.py` gains a second branch: `tts_recommended_runtime == "onnx-kokoro"` instantiates the new runtime; `kokoro` keeps the existing branch.

### Profiler recommendations

`_recommend_stt()` already returns `("onnx-whisper", "whisper-small", "cpu")` for Qualcomm NPU hosts. This slice extends it to return `("onnx-whisper", "whisper-small", "cpu")` for any ARM64 host whose readiness-backed evidence confirms `onnxruntime` is importable, independent of NPU presence. The existing fall-through behavior for x64 paths is preserved verbatim.

`_recommend_tts()` currently returns `("kokoro", "kokoro-v1.0", "cuda" | "cpu")` unconditionally. This slice extends it to return `("onnx-kokoro", "kokoro-v1.0-onnx", "cpu")` on ARM64 hosts and to preserve the existing tuple on x64 hosts.

### Evidence tokens

`resolve_backend_evidence_tokens("stt", matched_manifest_ids)` in `profiler.py` gains an ARM64 branch that returns `["import:onnxruntime", "stt_runtime:onnx-whisper"]` when `hw-arm64-base` is in `matched_manifest_ids`. Existing x64/CUDA/NPU-present token sets remain unchanged.

`resolve_backend_evidence_tokens("tts", ...)` gains `["import:kokoro_onnx", "tts_runtime:onnx-kokoro"]` on ARM64.

`derive_stt_device_readiness()` and `derive_tts_device_readiness()` in `preflight.py` gain ARM64-aware CPU-ready token sets so `selected_device` resolves to `"cpu"` with `selected_device_ready: True` when the ARM64 tokens verify. The existing CUDA-readiness logic is untouched.

### Wake word stays where it is

`pvporcupine>=3.0` already ships `win_arm64` wheels and is already in universal `backend/requirements.txt`. `PorcupineWakeWord` in `backend/app/runtimes/wake/local_runtime.py` requires no change. A valid ARM64-compiled `.ppn` file is user-supplied via `PVPORCUPINE_MODEL_PATH`; when absent or platform-mismatched, the existing Slice 5B degraded-mode contract applies — wake-unavailable falls back to push-to-talk, the degraded-conditions surface records it, and the resident loop continues.

This slice explicitly does not introduce new wake word code paths. Wake is left in its validated architectural position.

## Runtime Prerequisites

1. Windows ARM64 host with Python for ARM64 installed (CPython 3.11, 3.12, or 3.13 — the versions with `win_arm64` wheels on PyPI for `onnxruntime` and `kokoro-onnx`).
2. `backend/.venv` created with the ARM64 Python interpreter. All `backend/.venv/Scripts/python` commands continue to apply.
3. Ollama (or the chosen LLM runtime) must be reachable. LLM readiness is not extended by this slice; the current LLM runtime selector is consumed unchanged.
4. Internet connectivity for the first model acquisition run (Whisper ONNX model, `kokoro-v1.0.onnx`, `voices-v1.0.bin`); offline reruns must succeed against cached artifacts.

## Key Design Decisions

**Additive-only manifest model preserved.** No new merge semantics. `hw_arm64_base` adds packages; it never excludes or replaces. This keeps the provisioning correction closeout intact and avoids reopening it.

**Two parallel runtimes per family, not a replacement.** `FasterWhisperSTT` and `KokoroTTSRuntime` remain the x64/CUDA path. `OnnxWhisperSTT` and `KokoroOnnxTTSRuntime` are the ARM64 path. Selection is driven by profiler recommendations consuming readiness-backed evidence — the same boundary Slice 0.0 established.

**CPU-EP only.** This slice does not touch QNN, NPU, DirectML, or WebGPU. ARM64 acceptance is defined as "voice-capable on CPU-EP," not "voice-capable on NPU." Slice 0.0.B is the incremental NPU uplift.

**Wake word is not reinvented.** `pvporcupine` is already ARM64-capable, already in base, already validated. The existing degraded-mode path (wake unavailable → PTT) is the honest behavior when an ARM64 `.ppn` is not supplied, and is preserved.

**Runtime selection cache must not cross runtime families.** When extending `_STT_RUNTIME_CACHE` / `_TTS_RUNTIME_CACHE` keying, the runtime identity must be part of the key. Two different runtimes with the same model name and device string must not collide in the cache.

**ARM64 live tests skip cleanly on x64.** The x64/CUDA regression host runs `scripts/validate_backend.py --scope all` and sees a clean PASS with ARM64-specific tests skipped for host-class reasons, not errors. This keeps the regression gate usable from either host.

**The turn engine, personality, memory, and session service are consumed, not touched.** Every layer above `STTBase` / `TTSBase` is inherited from proven slices. Slice 0.0.A is a runtime-availability slice, not a behavioral slice.

## Risks / Sequencing Notes

- `kokoro-onnx` requires `kokoro-v1.0.onnx` and `voices-v1.0.bin` which are not HuggingFace repo files in the same shape as the existing `kokoro` model. Model catalog extension must handle the file-pair pattern honestly; do not conflate with the `kokoro` entry.
- ONNX Whisper model selection should start at `whisper-small` to match the current Qualcomm recommendation. Larger models may be added later but are not required for acceptance.
- CPU-EP latency on ARM64 is higher than CUDA on x64. Acceptance is voice-capable, not voice-realtime-parity. Do not set unrealistic latency gates.
- `espeak-ng` may be a runtime dependency of the phonemizer path `kokoro-onnx` uses. If the ONNX pipeline avoids external phonemizer calls, this is moot; if it does not, the dependency must be documented in the manifest notes but not silently introduced.
- The profiler's Qualcomm-NPU-implies-laptop classification remains correct. This slice does not change `classify_device_class()`.
- `.ppn` keyword files are platform-specific. Agents must not attempt to reuse an x64 `.ppn` on ARM64. The existing degraded-mode path is the correct behavior when the file is missing or platform-mismatched.
- Any profile-resolver change that tightens `arch` matching semantics must preserve the existing x64 `"amd64"` matching behavior exactly.
- Validation must not accept "packages imported" as proof. Acceptance requires a live voice turn completing on the ARM64 host with a non-empty transcript and a rendered audio artifact.
