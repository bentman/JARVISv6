## Slice 0.0.B — Qualcomm NPU Acceleration (QNN Uplift on ARM64 CPU-EP Baseline)

**Date:** 2026-04-18
**Status:** Proposed (do not start until Slice 0.0.A is Verified)
**Depends on:** Slice 0.0.A ARM64 Voice-Capable Path (Verified), Slice 0.0 Hardware Profile Provisioning (Verified), Slice 6 Durable Desktop Host (Verified)

---

## Slice 0.0.B Goal

Add Qualcomm NPU acceleration on top of the Slice 0.0.A CPU-EP baseline, without altering the baseline path.

Slice 0.0.A leaves the repo in a state where ARM64 hosts run a complete voice turn through `OnnxWhisperSTT` (CPU-EP) and `KokoroOnnxTTSRuntime` (CPU-EP). This slice adds one additional execution-provider choice — Qualcomm QNN via `onnxruntime-qnn>=2.0.0` — as a readiness-backed accelerated alternative for STT on Qualcomm hosts. Selection is driven by the same readiness rail Slice 0.0 established; when QNN readiness verifies, it is selected; when it does not, the Slice 0.0.A CPU-EP baseline is selected unchanged.

The completed experience after Slice 0.0.B:

```
ARM64 Qualcomm host:
  profiler detects Qualcomm NPU
  preflight resolves hw-arm64-base + hw-npu-present + hw-npu-qualcomm-qnn
  additive packages: onnxruntime + kokoro-onnx + onnxruntime-qnn
  readiness checks:
    - QNN EP registers and a quantized whisper context binary is loadable → QNN-ready
    - CPU-EP remains proven-ready from Slice 0.0.A
  STT selection:
    - QNN-ready → OnnxWhisperSTT(device="qnn")
    - QNN unverified → OnnxWhisperSTT(device="cpu")  [baseline]
  TTS remains CPU-EP (kokoro-onnx)
  live voice turn completes using QNN for STT, CPU-EP for TTS
  x64/CUDA rail unchanged
  ARM64 non-Qualcomm CPU-EP rail unchanged
```

**What Slice 0.0.B is not:** a rewrite of Slice 0.0.A, a new runtime family, a TTS acceleration slice, a replacement for `OnnxWhisperSTT`, a change to wake word or interruption behavior, a change to Slice 6, or a DirectML/WebGPU exploration. It is a bounded uplift that adds one device option behind existing selector contracts.

---

## Why this slice exists separately

QNN acceleration carries four concrete integration concerns that are not present in the CPU-EP baseline:

1. **QNN EP is a plugin execution provider.** `onnxruntime-qnn>=2.0.0` ships as a separate wheel that registers a `QNNExecutionProvider` against a standard `onnxruntime` install. Its presence must be detectable and its registration must be verified before device selection.
2. **HTP models must be quantized.** The Qualcomm HTP backend only accepts quantized models. A float32 Whisper ONNX model runs on CPU-EP, not QNN. This means the QNN path requires a second model artifact (a pre-quantized Whisper ONNX or a precompiled QNN context binary), not the same model the CPU-EP baseline uses.
3. **QAIRT SDK native libraries live outside `backend/.venv`.** `QnnHtp.dll` (or equivalent backend `.dll` for other QNN backends) is supplied by the Qualcomm QAIRT SDK and must be discoverable at session-creation time. This is architecturally similar to the Windows CUDA DLL bootstrap the repo already owns in `backend/app/hardware/preflight.py`, and belongs in the same owner.
4. **Kernel coverage is a subset of ONNX.** Not every ONNX op is implemented on HTP. Session creation can succeed with a valid backend path but fall back to CPU during graph partitioning, or fail entirely on unsupported ops. Readiness verification must probe actual session creation with the target model, not just package import.

Slice 0.0.A deliberately deferred these concerns because they are independent of making ARM64 voice-capable. They belong in their own scoped slice so CPU-EP remains the stable, always-selectable baseline while QNN is layered on.

## Scope

This slice covers:

- one new hardware manifest: `config/hardware/hw_npu_qualcomm_qnn.json`, gated on Qualcomm NPU presence, additive with `onnxruntime-qnn`
- extension of `backend/app/hardware/preflight.py` to own QNN-specific DLL/backend-path bootstrap symmetrically with the existing Windows CUDA DLL bootstrap
- new evidence-token types in `backend/app/hardware/profiler.py::resolve_backend_evidence_tokens()` for QNN (package import, EP registration, backend-path existence, session-create probe against a QNN-targeted model)
- extension of `derive_stt_device_readiness()` in preflight to recognize `qnn` as a device alongside `cuda` and `cpu`
- extension of `OnnxWhisperSTT` (from Slice 0.0.A) to accept `device="qnn"` and construct sessions with the `QNNExecutionProvider` + `provider_options.backend_path`
- a quantized Whisper ONNX catalog entry in `config/models/stt.yaml` (QNN-targeted model, distinct from the Slice 0.0.A CPU-EP model)
- unit coverage for QNN readiness derivation, evidence tokens, and device-selection behavior under `backend/tests/unit/test_hardware_readiness.py`
- a QNN-gated live acceptance test that skips cleanly when QNN readiness is not present

## Out of scope

This slice does not:

- modify `hw_arm64_base.json`, `hw_npu_present.json`, `hw_x64_base.json`, or any other existing manifest
- modify `KokoroOnnxTTSRuntime` or add QNN acceleration to TTS
- replace `OnnxWhisperSTT` with a different runtime — QNN is a new `device` value inside the existing runtime
- modify the additive-only manifest merge model
- modify the x64/CUDA STT/TTS rail
- modify Slice 6 desktop/session surfaces, `ResidentSessionService`, backend API routes, or cognition/memory/interruption behavior
- require QNN readiness for ARM64 acceptance — the Slice 0.0.A CPU-EP path remains the honest fallback on any Qualcomm host where QNN evidence does not verify
- install or configure the QAIRT SDK itself — SDK installation is an operator/user prerequisite, documented via manifest notes and `.env.example`; it is not a package-manager-installable artifact and must not be provisioned by preflight
- extend QNN to AMD GPU, Intel NPU, Apple NPU, or other vendor accelerators (each would be a separate manifest + device option)

## Architectural Approach

### QNN arrives as a third device option in the existing STT runtime

`OnnxWhisperSTT` already accepts `device` in its constructor (Slice 0.0.A). This slice adds `"qnn"` to the accepted values. Session construction becomes device-aware:

- `device == "cpu"` — existing CPU-EP session (Slice 0.0.A, unchanged)
- `device == "qnn"` — `InferenceSession(model_path, providers=["QNNExecutionProvider"], provider_options=[{"backend_path": <resolved QnnHtp.dll>}])`, loading a QNN-targeted quantized model

Session construction failures on `qnn` raise `RuntimeError("OnnxWhisperSTT: transcription failed (<reason>)")` — same fail-closed contract as every other runtime. No silent fallback to CPU inside the runtime. Fallback behavior is a readiness-derivation concern, not a runtime concern: if QNN readiness does not verify, the selector chooses `device="cpu"` before the runtime is instantiated.

### QAIRT SDK native-library discovery lives in preflight

`backend/app/hardware/preflight.py` already owns Windows CUDA DLL directory bootstrap. A symmetric QNN bootstrap is added:

- `ensure_windows_qnn_backend_path_bootstrap()` — a new public function paralleling `ensure_windows_cuda_dll_bootstrap()`
- resolves `QnnHtp.dll` (or the configured backend `.dll`) via a short priority list:
  1. an explicit environment variable `QAIRT_BACKEND_PATH` if set
  2. a standard QAIRT SDK install location derived from an env var `QNN_SDK_ROOT`
  3. an operator-documented custom path
- registers the containing directory via `os.add_dll_directory()` (same mechanism as CUDA)
- exposes the resolved backend DLL path for the runtime to consume as `provider_options.backend_path`

The CUDA bootstrap remains untouched. The two functions are peers.

### Evidence tokens for QNN

New token shapes recognized by `_verify_evidence_token()` in preflight:

- `"import:onnxruntime_qnn"` — reuses the existing `import:` shape
- `"qnn_backend_path:QnnHtp.dll"` — a new token type that resolves the backend DLL via the QNN bootstrap owner and returns `ok: True` only when the resolved path exists on disk
- `"qnn_session_probe:<model_identifier>"` — a new token type that attempts `InferenceSession(...)` against the QNN-targeted catalog model with `QNNExecutionProvider`; returns `ok: True` only when the session constructs without exception

The third token is the honest readiness gate. Package presence and DLL existence are necessary but not sufficient; a QNN session that cannot partition the target model is not QNN-ready.

### Readiness derivation extension

`derive_stt_device_readiness()` in preflight gains a QNN preference layer that runs before CPU-ready evaluation on Qualcomm hosts:

```
precedence (Qualcomm host):
  1. QNN-ready (manifest matches, all QNN tokens pass)    → selected_device = "qnn"
  2. CPU-ready (Slice 0.0.A tokens pass)                  → selected_device = "cpu"
  3. unavailable

precedence (non-Qualcomm ARM64 host):
  1. CPU-ready                                            → selected_device = "cpu"
  2. unavailable

precedence (x64/NVIDIA host, unchanged):
  1. CUDA-ready                                           → selected_device = "cuda"
  2. CPU-ready                                            → selected_device = "cpu"
  3. unavailable
```

The existing CUDA-vs-CPU logic on x64 is untouched. The QNN branch is added as a new top-priority device for Qualcomm hosts only, behind manifest-match gating. This keeps the derivation shape consistent with the pattern Slice 0.0 established.

### Profiler recommendations

`_recommend_stt()` continues to return `("onnx-whisper", "whisper-small", ...)` on ARM64 hosts. The device field becomes readiness-driven rather than hardcoded: the profiler emits the recommendation tuple with its default (`"cpu"`), and `run_profiler()`'s existing readiness-override mechanism (same pattern as x64/CUDA) upgrades `stt_recommended_device` to `"qnn"` when preflight derives it.

The STT model recommendation on Qualcomm hosts is extended to prefer the QNN-targeted quantized model (`whisper-small-onnx-qnn` or equivalent — exact catalog name determined at implementation time) when QNN readiness verifies, and the Slice 0.0.A CPU-EP model (`whisper-small-onnx`) otherwise. The recommendation tuple carries the model identity explicitly; runtime selection consumes it verbatim.

### TTS is unchanged

No QNN-accelerated TTS path is introduced. `KokoroOnnxTTSRuntime` remains CPU-EP. QNN TTS could be a future slice; it is not this one.

### Wake word is unchanged

No change to `pvporcupine` or wake-word runtime behavior. Same degraded-mode contract as Slice 5B and Slice 0.0.A.

## Runtime Prerequisites

All Slice 0.0.A prerequisites carry forward. Additionally:

1. **Qualcomm QAIRT SDK installed on the host** (user/operator prerequisite, not pip-installable). The path to the backend DLL is exposed to preflight via `QAIRT_BACKEND_PATH` or derivable from `QNN_SDK_ROOT`. Both are documented in `.env.example`.
2. **A QNN-targeted quantized Whisper ONNX model** available for acquisition through the existing catalog/manager path. The model identity is captured in `config/models/stt.yaml` and acquired by `scripts/ensure_models.py` the same way the CPU-EP model is.
3. **`onnxruntime-qnn>=2.0.0`** becomes an additive package under `hw_npu_qualcomm_qnn.json`. It does not enter `backend/requirements.txt` and does not enter `hw_arm64_base.json`.

## Key Design Decisions

**QNN is a device, not a new runtime family.** `OnnxWhisperSTT` owns both CPU-EP and QNN-EP session construction, differentiated by `device`. Adding a parallel `QnnWhisperSTT` class would duplicate provider-neutral orchestration and create two runtime identities that need equivalent coverage. One class with device-aware session construction is the smaller, more durable choice.

**QAIRT SDK is an operator prerequisite, not a preflight install target.** `onnxruntime-qnn` is pip-installable; the QAIRT SDK is not. Preflight must not attempt to install or relocate SDK binaries. Its role is to resolve, verify, and register the DLL directory — the same pattern it already owns for CUDA on Windows.

**Session-probe evidence is the honest readiness gate.** Package presence and DLL existence can both be true while the target QNN model fails to partition on HTP due to op coverage. The session-probe token makes this failure visible at readiness time, not at the first live voice turn. This preserves the fail-closed contract across the readiness rail.

**CPU-EP is the honest fallback on Qualcomm hosts.** When QNN readiness does not verify for any reason — missing SDK, wrong DLL version, unsupported op in the target model — the Slice 0.0.A CPU-EP path must select cleanly. This is not a degraded mode, it is the baseline. Operators should see `selected_device: cpu` on Qualcomm hosts without QAIRT as a successful outcome, not a failure.

**TTS remains CPU-EP in this slice.** QNN TTS would add a second quantized-model path, a second evidence chain, and a second readiness branch. None of it is required for voice-capable acceleration on ARM64. Deferring keeps this slice narrow and auditable.

**Additive-only manifest model preserved.** Three additive manifests can match on a Qualcomm ARM64 host (`hw-cpu-base`, `hw-arm64-base`, `hw-npu-present`, `hw-npu-qualcomm-qnn`) and compose cleanly. No new merge semantics. No replacement. No exclusion.

**Readiness-override in `run_profiler()` is the single recommendation-elevation surface.** The existing pattern — profiler emits a tuple with a default device, readiness derivation upgrades the device when evidence verifies — is reused unchanged. No parallel authority contract.

**Live acceptance is gated by host reality, not by configuration.** The QNN live test skips based on what the profiler + preflight actually derive, not based on an environment flag. This keeps the test honest: if QNN is truly ready, the test must pass; if not, the test must skip; no third state.

## Risks / Sequencing Notes

- QAIRT SDK versions, `QnnHtp.dll` ABIs, and `onnxruntime-qnn` versions have interlocking compatibility constraints. Manifest pinning must be chosen from confirmed-compatible combinations at implementation time; the `>=2.0.0` floor in this doc is a minimum, not a guarantee.
- HTP op coverage changes over `onnxruntime-qnn` versions. The session-probe token is the correct defense — it reflects the actual coverage at the current combination of EP version + model.
- Quantized Whisper ONNX artifacts are not interchangeable with the float CPU-EP model. Catalog entries must be distinct, and `scripts/ensure_models.py` must acquire them as separate models. Do not attempt to reuse the Slice 0.0.A model directory.
- QNN EP registration on `onnxruntime>=1.20` uses the plugin EP mechanism (`register_execution_provider_library()`) rather than the legacy built-in EP path. The runtime branch must follow the version's supported registration pattern.
- `OnnxWhisperSTT` cache-key handling must include the runtime family and device explicitly. A CPU-EP session and a QNN-EP session of the same model name are not the same object and must not collide.
- Any change to `_recommend_stt()` that alters the default device for Qualcomm hosts before readiness verification must not regress the Slice 0.0.A CPU-EP live acceptance on hosts where QNN readiness does not verify.
- The existing Qualcomm-NPU-implies-laptop classification in `classify_device_class()` remains correct and must not be altered by this slice.
- Validation must not accept "QNN EP registers" or "backend DLL exists" as proof of QNN readiness. Only the session-probe plus the live voice turn constitute acceptance.
- If no QNN-ready Qualcomm ARM64 host is available for live testing, the slice is not complete. Partial unit coverage with no live proof is not acceptance under the `ProjectVision.md` live-runtime-validation principle.
- The x64/CUDA host has no reason to exercise QNN code paths at all; the regression gate on x64 must show the QNN manifest does not match, evidence tokens are not emitted, and readiness derivation is identical to pre-slice behavior.
