## Slice 5A Goal

Give the verified backend a single, honest, user-facing entry point.

Everything from Slices 0–4 works correctly in the real runtime. What does not yet exist is a place for a user to actually run the assistant — a canonical start path, a typed fallback alongside voice, visible state and readiness, and a minimal but real conversation surface.

Slice 5A is not a UI-first slice. It is the smallest application shell that makes the already-proven runtime accessible without test scripts.

The completed application loop after Slice 5A:

```
user runs one command / launches one entry point
→ hardware profiled, readiness checked, personality loaded
→ startup summary printed (profile, readiness, degraded conditions)
→ session opened
→ shell accepts: 'v' to run a voice turn, '<typed text>' for text turn, 'q' to quit
→ each turn executes through the existing voice_service / turn_service stack
→ interruption, continuity, and artifact behavior inherited from proven runtime
→ state visible throughout: [STATE] transitions printed
→ degraded conditions surfaced before the user wastes a turn
→ session closed cleanly on exit
```

**What Slice 5A is not:** a GUI, a tray app, a wake-word listener, or a production-ready desktop shell. That is Slice 5B.

---

## Architectural Approach

### No new runtime surfaces

The entire voice/STT/TTS/LLM/interruption stack is proven and stays untouched. Slice 5A wraps it with a minimal interactive layer — a shell loop and a startup sequence — rather than replacing or reimplementing any runtime behavior.

### Text turns use `run_turn()` directly

`run_turn()` in `backend/app/services/turn_service.py` already accepts typed input and runs through the same cognition path as voice. Slice 5A wires this as the text fallback without creating any parallel execution path.

### Personality-driven Kokoro voice selection

`PersonalityProfile` already carries `voice_pacing` and `voice_energy` fields. The `KokoroTTSRuntime` accepts a `voice` parameter. Slice 5A maps personality fields to Kokoro voice IDs using a simple deterministic resolver so the spoken voice reflects the configured personality without complex adaptive logic.

### Startup sequence owns readiness

`bootstrap_readiness.py` already exists and covers STT readiness. Slice 5A calls it (or equivalent readiness probe) as part of startup, surfaces the result to the user, and gates voice-turn availability on STT readiness. TTS readiness is checked separately. LLM reachability is checked (Ollama `is_available()`). All degraded conditions are printed before the first turn.

## Runtime Prerequisites

All prior slice prerequisites carry forward. Additionally:

1. **`scripts/run_jarvis.py` is the canonical entry point** — no other entry path should be defined in this slice.

## Sub-Slice 5A.1 — Startup Sequence and Readiness Summary

### Objective

Create a callable startup sequence that profiles hardware, checks readiness, loads personality, and returns a named summary structure. The summary is printed at startup and used to gate voice/TTS availability in the shell loop.

### Files to Create

**`backend/app/services/startup_service.py`**

```python
@dataclass
class StartupSummary:
    profile_id: str
    device_class: str

    stt_ready: bool
    stt_selected_device: str       # "cuda" | "cpu" | "unavailable"
    tts_ready: bool
    tts_selected_device: str
    llm_ready: bool
    llm_runtime_name: str          # e.g. "OllamaLLM" or "unavailable"

    personality_profile_id: str
    personality_display_name: str
    voice_name: str                # resolved Kokoro voice ID

    degraded_conditions: list[str]  # human-readable list; empty when fully ready


def run_startup(*, personality_name: str = "default") -> tuple[StartupSummary, FullCapabilityReport, PersonalityProfile]:
    """Profile hardware, check readiness, load personality.
    Returns (summary, report, personality).
    Never raises on degraded conditions — records them in summary.degraded_conditions instead.
    Raises only on catastrophic failures (missing personality file, profiler crash).
    """
```

Implementation rules:
- calls `run_profiler()` for the `FullCapabilityReport` (already includes `BackendReadiness`)
- checks `report.readiness.stt_selected_device` and `report.readiness.tts_selected_device` for readiness state
- checks `OllamaLLM().is_available()` for LLM reachability; records name if available
- calls `load_personality_profile(personality_name)` — fails closed with `FileNotFoundError` if missing
- calls `resolve_voice_from_personality(personality)` (see 5A.2) for the Kokoro voice ID
- populates `degraded_conditions` with one plain-English string per degraded path

**`backend/app/services/startup_service.py`** — also exports:

```python
def print_startup_summary(summary: StartupSummary) -> None:
    """Print a clear, structured startup summary to stdout."""
```

Example output:
```

║  JARVIS — Local Voice Assistant              ║
╚══════════════════════════════════════════════╝
  Profile:   nvidia-cuda-desktop-63gb (desktop)
  STT:       faster-whisper (cuda)   ✓
  TTS:       kokoro (cpu)            ✓  voice: bf_isabella
  LLM:       OllamaLLM (phi3:mini)  ✓
```

## Sub-Slice 5A.2 — Personality-to-Voice Resolver

### Objective

Map `PersonalityProfile` fields deterministically to a Kokoro voice ID. This is the only per-slice change to the existing personality/TTS surface.

### Files to Create

**`backend/app/personality/voice_resolver.py`**

```python
def resolve_voice_from_personality(personality: PersonalityProfile) -> str:
    """Return a Kokoro voice ID appropriate for the personality profile.

    Mapping rules (deterministic, no model required):
    - voice_energy=calm + formality=professional → bf_isabella (composed British female)
    - voice_energy=calm + warmth=high            → af_sarah (warm American female)
    - voice_energy=moderate + assertiveness=high → am_adam (direct American male)
    - voice_energy=high                          → am_michael (energetic American male)
    - default fallback                           → bf_isabella

    Returns a valid voice ID string from config/models/tts.yaml supported_voices.
    """
```

Rules:
- pure function, no I/O
- returns a voice ID string only
- default fallback is always a valid voice (never returns None or raises)
- the mapping is intentionally simple and inspectable — one conditional chain, no ML

## Sub-Slice 5A.3 — Text Turn Shell Integration

### Objective

Add a typed-text turn path to the application shell that uses `run_turn()` directly — the same cognition path as voice — without creating a parallel execution path.

### Files to Create

**`backend/app/services/text_service.py`**

```python
@dataclass
class TextTurnResult:
    response: str
    failed: bool = False
    failure_reason: str | None = None


def run_text_turn(
    text: str,
    report: FullCapabilityReport,
    personality: PersonalityProfile,
    *,
    session: Session | None = None,
    memory: WorkingMemory | None = None,
) -> TextTurnResult:
    """Route typed text through the same turn_service cognition path as voice.
    Returns TextTurnResult. Never raises — failures are captured in the result.
    """
```

Implementation rules:
- calls `run_turn(report, personality, text, session=session, memory=memory, input_modality="text")`
- wraps any exception in `TextTurnResult(response="", failed=True, failure_reason=str(exc))`
- prints `[TEXT TURN] <text>` before execution and `[RESPONSE] <response>` after, matching the voice path print pattern
- does not call STT, TTS, or any audio path

## Sub-Slice 5A.4 — Canonical Application Entry Point

### Objective

Create the single, supported way to run the assistant locally. This is the first real "you can just run JARVIS now" surface.

### Files to Create

**`scripts/run_jarvis.py`**

```
usage: backend/.venv/Scripts/python scripts/run_jarvis.py [--personality <name>] [--turns <n>]

Shell loop behavior:
  [q / quit / exit]  → close session and exit
  [v / voice]        → run one voice turn (gated on stt_ready)
  [<any other text>] → run one text turn

Turn loop:
  - each voice turn calls run_voice_turn(report, personality, session=session, memory=memory)
  - each text turn calls run_text_turn(text, report, personality, session=session, memory=memory)
  - interruption result (result.interrupted=True) prints [INTERRUPTED] and loops back without closing session
  - FAILED state prints [FAILED] and loops back (session preserved)
  - loop exits cleanly on quit or --turns exhaustion
```

Implementation rules:

```python
def main(argv=None) -> int:
    # 1. Parse args
    # 2. run_startup() → summary, report, personality
    # 3. print_startup_summary(summary)
    # 4. If no readiness at all (stt_ready=False AND tts_ready=False AND llm_ready=False):
    #    print error, return 1
    # 5. Open session + WorkingMemory
    # 6. Shell loop:
    #    - print "[IDLE] > " prompt
    #    - read line
    #    - dispatch voice / text / quit
    #    - print turn result
    # 7. Close session, return 0
```

**Important:** the voice turn is gated on `summary.stt_ready`. If STT is not ready and the user types `v`, print:
```
[DEGRADED] Voice unavailable (STT not ready) — enter text instead
```

**Session continuity:** voice and text turns share the same `session` and `memory` objects so multi-turn context works across modalities within one shell session.

**Degraded TTS:** if TTS is unavailable and a voice turn completes, the response is printed as text. This already works via the existing `[DEGRADED] TTS unavailable — text response only` path in `voice_service.py`.

## Key Design Decisions

**One entry point, two modalities, one session.** Voice and text turns share session and memory within one `run_jarvis.py` invocation. Context persists across modality switches within a session, which is already proven by `run_turn()` accepting `input_modality`.

**`startup_service.py` never raises on degraded conditions.** It captures readiness failures and surfaces them as strings in `degraded_conditions`. The application entry point decides whether to exit or continue in degraded mode. Catastrophic failures (missing personality file, profiler crash) still raise — those are not degraded conditions, they are repo configuration errors.

**Voice is gated on `stt_ready`.** If STT is not verified ready, typing `v` produces a clear degraded message rather than a timeout or a silent failure. This matches `ProjectVision.md` principle 7 (clear failure states).

**`voice_resolver.py` is a pure function.** No I/O, no model calls. The personality-to-voice mapping is a deterministic conditional chain. It can be extended in Slice 5B without touching any runtime code.

**Text turns do not call TTS.** Text input → cognition → text output. If the user wants spoken response for a text turn, that is a future feature. Keeping text turns text-only preserves the single-turn boundary and avoids adding TTS coupling to the text path in this slice.
