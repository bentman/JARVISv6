## Slice 5A Goal

Establish an application shell contract and an honest start path around the verified runtime core.

Everything from Slices 0–4 works correctly in the real runtime, but only through test scripts. What the repo is missing is:

- unified startup ownership that profiles hardware, verifies readiness, and loads personality in a single callable sequence
- a shared session surface that voice and text turns can both enter without separate invocation paths
- readiness and degraded-mode visibility before the first turn runs
- a shell-facing contract that a later desktop or frontend host can consume without redesign

Slice 5A delivers these by defining the backend service contract and proving it through a **bounded local shell adapter** (`scripts/run_jarvis.py`). That script is an implementation artifact for this slice — the minimal host needed to validate the contract. It is **not** the long-term application surface for a desktop-first, voice-first assistant. The durable output of this slice is the service contract (`startup_service.py`, `text_service.py`, `voice_resolver.py`), not the shell script itself.

`ProjectVision.md` defines JARVISv6 as desktop-first. A Python stdin loop is not the desktop shell — it is the simplest host that can exercise and validate the start-path contract before a real application host is built.

The completed start-path contract after Slice 5A:

```
shell host (any) calls run_startup()
→ hardware profiled, readiness checked, personality loaded
→ StartupSummary emitted: profile, STT/TTS/LLM readiness, degraded conditions
→ session opened; WorkingMemory initialized
→ host dispatches voice turns → run_voice_turn() [existing, proven]
→ host dispatches text turns → run_text_turn() [new, same cognition path]
→ interruption, continuity, and artifact behavior inherited from proven runtime
→ session closed cleanly on exit
→ host surfaces state ([STATE] transitions, degraded conditions) to user
```

**What Slice 5A is not:** a GUI, a tray app, a wake-word listener, or a durable production desktop shell. Those come in Slice 5B and beyond.

---

## Architectural Approach

### The shell contract, not the shell

The primary output is `backend/app/services/startup_service.py` — a callable startup sequence that any host (script, desktop app, API server, future Tauri shell) can invoke. The local shell adapter in `scripts/run_jarvis.py` is the first concrete host exercising this contract, not the intended final host.

### No new runtime surfaces

The entire voice/STT/TTS/LLM/interruption stack is proven and stays untouched. Slice 5A adds the startup and text-turn service layer that sits above it.

### Text turns use `run_turn()` directly

`run_turn()` in `backend/app/services/turn_service.py` already accepts typed input through the same cognition path as voice. `text_service.py` wraps it with a clean, host-agnostic result type — no parallel execution path is created.

### Personality-driven voice selection

`PersonalityProfile` already carries `voice_pacing` and `voice_energy`. Slice 5A introduces `voice_resolver.py` to map these to Kokoro voice IDs deterministically. That resolver is then consumed by the startup summary and reusable by any future host.

### Startup sequence owns readiness

`BackendReadiness` from `run_profiler()` already carries STT/TTS readiness state. Slice 5A introduces `startup_service.py` to read it, check LLM reachability, and return a `StartupSummary` with explicit `degraded_conditions`. The local shell adapter and any future host then consume this summary.

---

## Runtime Prerequisites

All prior slice prerequisites carry forward. Additionally:

1. **`.env` may carry `OLLAMA_MODEL` and `OLLAMA_BASE_URL`** — startup reads these and prints resolved values.
2. **`config/personality/default.yaml` + `config/personality/jarvis_personality.json`** — both must be present; `run_startup()` fails closed with a named error if either is missing.
3. **`scripts/run_jarvis.py` is the bounded local shell adapter for this slice** — it validates the start-path contract but does not represent the intended steady-state application host.

## Sub-Slice 5A.1 — Startup Service and Readiness Summary

### Objective

Create `startup_service.py` — a callable startup sequence that any host can invoke to get a complete `StartupSummary`. This is the primary contract deliverable of Slice 5A.

### Files to Create

**`backend/app/services/startup_service.py`**

```python
@dataclass
class StartupSummary:
    profile_id: str
    device_class: str

    stt_ready: bool
    stt_selected_device: str       # "cuda" | "cpu" | "unavailable"
    tts_ready: bool
    tts_selected_device: str
    llm_ready: bool
    llm_runtime_name: str          # e.g. "OllamaLLM" or "unavailable"

    personality_profile_id: str
    personality_display_name: str
    voice_name: str                # resolved Kokoro voice ID

    degraded_conditions: list[str]  # human-readable list; empty when fully ready


def run_startup(
    *, personality_name: str = "default"
) -> tuple[StartupSummary, FullCapabilityReport, PersonalityProfile]:
    """Profile hardware, check readiness, load personality.

    Returns (summary, report, personality) — usable by any host.
    Never raises on degraded conditions — records them in summary.degraded_conditions.
    Raises only on catastrophic failures: missing personality file, profiler crash.
    """


def print_startup_summary(summary: StartupSummary) -> None:
    """Print a clear, structured startup summary to stdout.
    Hosts that render their own UI can omit this and consume StartupSummary directly.
    """
```

Implementation rules:
- calls `run_profiler()` — emits `FullCapabilityReport` including `BackendReadiness`
- reads `report.readiness.stt_selected_device` / `tts_selected_device` for readiness state
- checks `OllamaLLM().is_available()` for LLM reachability
- calls `load_personality_profile(personality_name)` — raises `FileNotFoundError` if missing
- calls `resolve_voice_from_personality(personality)` (5A.2) for the Kokoro voice ID
- populates `degraded_conditions` with one plain-English string per degraded path

Example `print_startup_summary` output (for the bounded local shell; other hosts may render differently):
```
╔══════════════════════════════════════════════╗
║  JARVIS — Local Voice Assistant              ║
╚══════════════════════════════════════════════╝
  Profile:     nvidia-cuda-desktop-63gb (desktop)
  STT:         faster-whisper (cuda)   ✓
  TTS:         kokoro (cpu)            ✓  voice: bf_isabella
  LLM:         OllamaLLM (phi3:mini)  ✓
  Personality: Jarvis — professional with humor
══════════════════════════════════════════════
  ⚠  DEGRADED: TTS running on CPU (CUDA not ready)
══════════════════════════════════════════════
```

## Sub-Slice 5A.2 — Personality-to-Voice Resolver

### Objective

Introduce a deterministic mapping from `PersonalityProfile` fields to a Kokoro voice ID. Pure function, no I/O, reusable by any host.

### Files to Create

**`backend/app/personality/voice_resolver.py`**

```python
def resolve_voice_from_personality(personality: PersonalityProfile) -> str:
    """Return a Kokoro voice ID appropriate for the personality profile.

    Mapping rules (deterministic, no model required):
    - voice_energy=calm + formality=professional → bf_isabella (composed British female)
    - voice_energy=calm + warmth=high            → af_sarah (warm American female)
    - voice_energy=moderate + assertiveness=high → am_adam (direct American male)
    - voice_energy=high                          → am_michael (energetic American male)
    - default fallback                           → bf_isabella

    Returns a valid voice ID string from config/models/tts.yaml supported_voices.
    Never returns None or raises.
    """
```

## Sub-Slice 5A.3 — Text Turn Service

### Objective

Add `text_service.py` — typed-text input through the same cognition path as voice. Host-agnostic: returns a result type rather than depending on terminal output. Any host (script, API, desktop app) can call this and render the result in its own way.

### Files to Create

**`backend/app/services/text_service.py`**

```python
@dataclass
class TextTurnResult:
    response: str
    failed: bool = False
    failure_reason: str | None = None


def run_text_turn(
    text: str,
    report: FullCapabilityReport,
    personality: PersonalityProfile,
    *,
    session: Session | None = None,
    memory: WorkingMemory | None = None,
) -> TextTurnResult:
    """Route typed text through turn_service — same cognition path as voice.
    Returns TextTurnResult. Never raises — failures captured in result.
    Does not invoke STT, TTS, or any audio path.
    """
```

Implementation rules:
- calls `run_turn(report, personality, text, session=session, memory=memory, input_modality="text")`
- wraps exceptions in `TextTurnResult(response="", failed=True, failure_reason=str(exc))`
- terminal prints such as `[TEXT TURN] <text>` and `[RESPONSE] <response>` belong in the bounded shell adapter, not the core service contract

## Sub-Slice 5A.4 — Bounded Local Shell Adapter

### Objective

Create `scripts/run_jarvis.py` — the minimal host that exercises and validates the Slice 5A start-path contract. This script is an implementation artifact for this slice, not the intended long-term application surface.

Its purpose: prove that `run_startup()`, `run_text_turn()`, `run_voice_turn()`, session continuity, and degraded-mode visibility all work together in a real invocation context before a desktop or richer application host is built on top.

### Files to Create

**`scripts/run_jarvis.py`**

```
usage: backend/.venv/Scripts/python scripts/run_jarvis.py [--personality <name>] [--turns <n>]

Bounded shell loop (push-to-talk only in this slice; wake word added in 5B):
  [q / quit / exit]  → close session and exit
  [v / voice]        → run one voice turn (gated on stt_ready)
  [<any other text>] → run one text turn

Implementation:
  1. run_startup() → StartupSummary, report, personality
  2. print_startup_summary(summary)
  3. If stt_ready=False AND tts_ready=False AND llm_ready=False: print error, exit 1
  4. Open Session + WorkingMemory (shared across both modalities)
  5. Shell loop: dispatch by input
  6. Close session on exit
```

Key behaviors:
- voice turn gated on `summary.stt_ready`; if not ready, prints `[DEGRADED] Voice unavailable (STT not ready) — enter text instead`
- voice and text turns share the same `session` and `memory` — multi-turn context works across modalities
- `VoiceTurnResult.interrupted=True` prints `[INTERRUPTED]` and loops back without closing session
- `--turns 0` exits after startup summary (useful for CI/readiness validation)

### Validation Commands

```powershell

# One text turn non-interactively
echo "What is your name?" | backend/.venv/Scripts/python scripts/run_jarvis.py --turns 1
```

## Key Design Decisions

**The durable output is the service contract, not the script.** `startup_service.py` and `text_service.py` are the architectural deliverables. `scripts/run_jarvis.py` is a test harness for the contract. Any future desktop or frontend host replaces the script and consumes the same service contract.

**`startup_service.py` never raises on degraded conditions.** Degraded paths produce strings in `degraded_conditions`; the host decides what to show and whether to continue. Catastrophic failures (missing personality, profiler crash) raise — those are configuration errors, not degraded runtime states.

**Voice and text share one session.** Session and WorkingMemory are initialized once per shell invocation and passed to both `run_voice_turn()` and `run_text_turn()`. This is already supported by both service functions via `session=` and `memory=` keyword arguments.

**Voice is gated on `stt_ready`.** If STT is not verified ready, voice input is blocked with a named degraded message before any capture is attempted.

**Text turns do not call TTS.** Text → cognition → text response only. Spoken response for text turns is a future host-level decision, not wired here.

**`print_startup_summary` is host-optional.** A future desktop host can call `run_startup()` and render `StartupSummary` in its own UI without using the terminal print function.
