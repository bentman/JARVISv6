## Slice 6 Goal

Replace the bounded proving host (`scripts/run_jarvis.py`) with the real `desktop/` product shell and close the resident interaction gap.

Two concrete things are missing that this slice delivers:

1. **A durable application surface.** `scripts/run_jarvis.py` was always a temporary proving host. `ProjectVision.md` defines JARVISv6 as desktop-first. The assistant needs a real home: system tray, start/stop/running controls, conversation display, and clean process lifecycle — not a terminal loop.

2. **A resident session.** The current verified boundary is "single-turn with caller-initiated next turn." The user must manually re-invoke between every turn. A real assistant maintains a continuous session within a run — listen, respond, and listen again without the user doing anything between turns.

The completed experience after Slice 6:

```
user launches JARVIS (tray icon appears, status: starting)
→ startup runs: hardware profiled, readiness checked, personality loaded
→ status transitions to: ready / degraded (if applicable)
→ user says "hey Jarvis" or presses push-to-talk hotkey
→ LISTENING begins automatically
→ voice turn executes through existing proven runtime stack
→ response spoken, status returns to: listening (resident loop continues)
→ user can speak again without re-invoking
→ user clicks tray → Stop: session closes cleanly, tray shows stopped
→ all turn artifacts and session records written as before
```

**What Slice 6 is not:** a new voice/STT/TTS/cognition implementation. Every runtime behavior is proven and untouched. This slice adds the resident loop service and the desktop host shell that consumes the existing service contract.

---

## Architectural Approach

### Backend API as the host/service bridge

The `desktop/` shell is a native application (Tauri + WebView frontend). It cannot call Python directly. The bridge is a lightweight local FastAPI server — `backend/app/api/` is already declared in `repo_tree.md` with routes for voice, task, session, diagnostics, and health. Slice 6 brings these routes to working state so the desktop shell can call them over `localhost`.

The backend process runs as a sidecar launched by the desktop application. The desktop shell starts the backend, polls the health endpoint, and then drives interaction through the API. When the user stops the assistant, the desktop tears down the backend process cleanly.

This keeps the boundary clean: backend owns all orchestration, runtime selection, session state, and memory. Desktop owns only presentation, controls, and user-facing state.

### Resident session service

The resident loop lives in a new `backend/app/services/session_service.py` — a long-running session owner that:
- holds one active `Session` and `WorkingMemory` for the lifetime of a resident run
- exposes start/stop/status operations
- drives the voice capture → turn → response → resume cycle continuously
- surfaces session state so the API can report it to the desktop

`run_voice_turn()` and `run_text_turn()` are called from within this service unchanged. No runtime behavior changes.

### Desktop shell (Tauri)

`desktop/` is a Tauri application. The Tauri backend (Rust) manages:
- backend Python process lifecycle (spawn on start, kill on stop)
- system tray icon and menu
- global hotkey registration (push-to-talk)
- inter-process coordination

The Tauri frontend (WebView, HTML/CSS/JS) renders:
- conversation display (turn-by-turn transcript + response, rendered from backend-owned turn records)
- assistant status indicator (starting / listening / speaking / thinking / degraded / stopped)
- configuration display (active personality profile name and readiness summary — read-only; live profile mutation is not in scope for this slice)

Tauri is the natural choice given `repo_tree.md` already names it and it supports Windows natively.

### Wake word host integration

`PorcupineWakeWord` and `select_wake_runtime()` are already implemented (Slice 5B). The resident session service calls `select_wake_runtime()` at startup, runs detection in the background between turns, and triggers the next voice turn when the wake flag fires. The desktop shell does not own wake word detection — it only reflects the state (listening / wake-active).

## Sub-Slice 6.1 — Resident Session Service + Backend API Foundation

### Objective

Add the resident session service that owns a continuous session loop, and bring up the minimal backend API so the desktop shell has something to call.

### Files to Create / Modify

**`backend/app/services/session_service.py`**

```python
@dataclass
class ResidentSessionState:
    status: str              # "idle" | "listening" | "transcribing" | "thinking"
                             #   | "speaking" | "interrupted" | "degraded" | "stopped"
    session_id: str | None
    turn_count: int
    degraded_conditions: list[str]
    last_response: str | None
    last_transcript: str | None


class ResidentSessionService:
    """Coordinates resident session lifecycle, invocation triggers, and session-status projection
    for the lifetime of one assistant run.

    Holds a WorkingMemory instance for turn continuity within a run. Does not own memory write
    policy, episodic memory, or cross-session retrieval — those remain in the memory layer.
    Calls run_voice_turn() and run_text_turn() from the existing service layer unchanged.
    Thread-safe: start/stop/status may be called from any thread.
    """

    def start(
        self,
        report: FullCapabilityReport,
        personality: PersonalityProfile,
        summary: StartupSummary,
    ) -> None:
        """Open session, start resident loop in background thread."""

    def stop(self) -> None:
        """Signal stop, close session, join background thread."""

    def submit_text(self, text: str) -> None:
        """Enqueue a text turn (for typed fallback from desktop)."""

    @property
    def state(self) -> ResidentSessionState:
        """Current observable state — safe to call from any thread."""
```

Implementation rules:
- Background thread: loop that calls `run_voice_turn()` continuously when status is `listening`
- Wake flag from `select_wake_runtime()` triggers a voice turn; push-to-talk queues one via threading event
- On `VoiceTurnResult.interrupted=True`: transition to `listening` (no re-invocation needed — the loop continues)
- On any exception: transition to `degraded`, log failure reason, loop continues unless `stopped`
- `stop()` sets a stop event, joins the background thread, closes session via `SessionManager.close_session()`

**`backend/app/api/routes/health.py`**

```python
GET /health → {"status": "ok", "profile_id": str, "stt_ready": bool, "tts_ready": bool, "llm_ready": bool}
```

**`backend/app/api/routes/session.py`**

```python
POST /session/start   → {"session_id": str, "status": str}
POST /session/stop    → {"session_id": str | None, "status": "stopped"}
GET  /session/state   → ResidentSessionState as JSON
POST /session/text    → {"text": str} → {"queued": true}
```

**`backend/app/api/dependencies.py`** — shared `ResidentSessionService` singleton, `StartupSummary`, `FullCapabilityReport`, `PersonalityProfile` injected at app startup.

**`scripts/run_backend.py`**

```python

Usage: backend/.venv/Scripts/python scripts/run_backend.py [--host 127.0.0.1] [--port 8765]
"""
```

Calls `uvicorn.run("backend.app.api.main:app", host=..., port=..., log_level="warning")`. Port 8765 is the JARVISv6 default.

**`backend/requirements.txt`** — add to `# Web / API Core`:

## Key Design Decisions

**FastAPI over direct IPC.** The desktop shell and backend are separate processes. A local HTTP API is simpler, more debuggable, and more extensible than custom IPC. It also sets up the correct boundary for future remote or multi-device use. Port 8765 is bound to `127.0.0.1` only — no external exposure.

**`ResidentSessionService` owns the loop.** The backend service layer is the resident authority. The Tauri frontend only polls state and sends commands. No orchestration logic lives in Rust or the WebView.

**Resident loop continues after interruption.** When `VoiceTurnResult.interrupted=True`, the loop transitions back to `listening` without closing the session. This is the correct behavior per `ProjectVision.md` interruption policy — the session is preserved and the assistant is immediately ready again.

**Wake word and PTT are equivalent triggers.** Both set a threading event that the resident loop reads as "start a voice turn." The service does not distinguish between them after trigger — the turn executes identically either way.

**Tauri chosen for `desktop/`.** `repo_tree.md` already names it. It builds native Windows executables, supports system tray, global hotkeys, and WebView-based UI. The frontend can start as minimal HTML/JS and evolve toward a richer framework later without changing the Rust backend or Python service contract.

**`run_jarvis.py` is not deleted.** It remains a useful developer/diagnostic tool for running offline backend validation without the desktop host. It is demoted, not removed.
