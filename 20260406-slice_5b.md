## Slice 5B Goal

Make JARVIS feel present, characterful, and responsive around the application shell that Slice 5A established.

Slice 5A created a working, honest entry point. Slice 5B adds:
- wake word so the user does not need to type `v` to speak
- richer personality shaping controls beyond the current minimal YAML fine-tuning
- assistant acknowledgment behaviors (audible/visible "I'm listening" cues)
- conversational flow polish that improves the felt quality of interaction without changing the underlying runtime

The completed experience after Slice 5B:

```
user says "hey Jarvis" (or configured phrase)
→ wake word detected → shell transitions to LISTENING automatically
→ user speaks → full voice turn executes
→ Jarvis acknowledges at start of each response (tone-appropriate)
→ personality-consistent spoken voice used throughout
→ degraded conditions handled gracefully with personality-appropriate fallback messaging
→ personality profile editable via config without code changes
→ shell remains stable and honest at all times
```

**Scope note:** this slice does not add a graphical UI, system tray, hotkey daemon, or overlay window. Those are valuable but belong after presence/persona is working cleanly in the terminal shell.

## Architectural Approach

### Wake word: `pvporcupine` via `backend/app/runtimes/wake/`

The `repo_tree.md` already declares `backend/app/runtimes/wake/` with `base.py`, `local_runtime.py`, and `wakeword_runtime.py`. Slice 5B implements the first real content of those files using Picovoice Porcupine — the most robust, local-first, free-tier-available wake word engine for Python on Windows.

**Why Porcupine:** `pvporcupine` provides pre-built wake word models for common phrases including "Hey Jarvis" at no cost under the Picovoice free tier. It runs entirely locally, uses a tiny ONNX model, and integrates with `sounddevice` via a callback stream. No cloud calls, no subscription required for the free tier.

**Activation key:** Porcupine requires a free Picovoice Access Key (environment variable `PICOVOICE_ACCESS_KEY`). If absent, wake word degrades cleanly to push-to-talk. This is declared explicitly in `BackendReadiness` via a new `wake_ready` field (added in 5B.1).

### Acknowledgment: thin personality-aware response prefix

JARVIS already generates responses through the cognition path. Slice 5B adds a lightweight acknowledgment behavior — a brief, personality-consistent prefix before the main response — implemented as a new `generate_acknowledgment(personality)` function in the personality layer. The acknowledgment is spoken via TTS before the main response synthesis, giving the user immediate feedback that JARVIS heard them.

### Personality shaping: YAML schema extension

The current `default.yaml` covers the required fields. Slice 5B extends the personality schema with optional shaping fields:
- `acknowledgment_phrase_style` — `"minimal"` | `"standard"` | `"warm"` (drives the acknowledgment generator)
- `wake_response_sound` — `"none"` | `"tone"` | `"phrase"` (future hook; tone playback only in 5B)
- `voice_name_override` — optional explicit Kokoro voice ID to override the resolver

These are additive optional fields with defaults matching current behavior, so existing profiles remain valid.

## Runtime Prerequisites

All Slice 5A prerequisites carry forward. Additionally:

1. **`PICOVOICE_ACCESS_KEY` in `.env`** — required for wake word; absence triggers graceful degradation to push-to-talk. `.env.example` must document this.
2. **`pvporcupine` package** — additive to `backend/requirements.txt`. Not in base because it is only needed when wake word is enabled.
3. **"Hey Jarvis" keyword file or built-in keyword** — Porcupine includes "porcupine", "bumblebee", "alexa" as built-ins. "Hey Jarvis" requires a custom keyword file (free download from Picovoice console) OR use of the "jarvis" built-in keyword available in newer Porcupine versions. Slice 5B uses the built-in `"jarvis"` keyword if available, falling back to push-to-talk if it is not.
4. **Audio input stream stability** — simultaneous Porcupine stream + barge-in stream must work. Porcupine uses its own `sounddevice` input stream; this is the same pattern validated in Slice 4 for `BargeInDetector`.

## Sub-Slice 5B.1 — Wake Word Runtime

### Objective

Implement the `wake/` runtime layer declared in `repo_tree.md`. Prove that Porcupine can run concurrently with the existing audio stack.

### Files to Create / Modify

**`backend/app/runtimes/wake/base.py`**

```python
from abc import ABC, abstractmethod
import threading

class WakeWordBase(ABC):
    @abstractmethod
    def start(self, wake_flag: threading.Event) -> None:
        """Begin listening. Set wake_flag when keyword detected."""

    @abstractmethod
    def stop(self) -> None:
        """Stop listening and release resources."""

    @abstractmethod
    def is_available(self) -> bool:
        """True if the runtime can be used on this host."""

    @property
    @abstractmethod
    def failed(self) -> bool: ...

    @property
    @abstractmethod
    def failure_reason(self) -> str | None: ...
```

**`backend/app/runtimes/wake/local_runtime.py`**

`PorcupineWakeWord(WakeWordBase)`:
- constructor: `__init__(self, access_key: str, keyword: str = "jarvis")`
- `is_available()`: attempts `import pvporcupine`; returns `True` if importable and `access_key` is non-empty
- `start(wake_flag)`: opens `pvporcupine.Porcupine`, opens `sounddevice.InputStream` callback that feeds 512-frame PCM16 chunks to Porcupine; on keyword detection sets `wake_flag`
- `stop()`: closes stream and Porcupine handle cleanly
- failure contract mirrors `BargeInDetector`: `failed` / `failure_reason` never raises

Stream rule: Porcupine requires 16kHz mono int16 PCM at exactly 512-frame blocks. `sounddevice.InputStream` with `dtype="int16"`, `blocksize=512`, `samplerate=16000` satisfies this.

**`backend/app/runtimes/wake/wakeword_runtime.py`**

```python
def select_wake_runtime(access_key: str | None) -> WakeWordBase | None:
    """Return PorcupineWakeWord if available and key present, else None."""
```

**`backend/app/core/capabilities.py`** — add to `BackendReadiness`:
```python

```

**`backend/app/hardware/profiler.py`** — extend `_derive_backend_readiness` to probe wake readiness:

`_probe_wake_available()` is a lightweight `import pvporcupine` check — no Porcupine handle opened at probe time.

## Sub-Slice 5B.3 — Acknowledgment Generator

### Objective

Add a personality-aware acknowledgment generator. When JARVIS starts processing a turn, it plays or prints a brief acknowledgment so the user knows they were heard.

### Files to Create

**`backend/app/personality/acknowledgment.py`**

```python
def generate_acknowledgment(personality: PersonalityProfile) -> str | None:
    """Return a brief acknowledgment phrase appropriate for the personality.

    Returns None when style is 'minimal' (no spoken acknowledgment, rely on [STATE] prints).
    Returns a short phrase for 'standard' and 'warm' styles.

    Examples:
      standard: "Got it.", "Sure.", "One moment."
      warm:     "Of course.", "Absolutely.", "Right away."

    Phrases are selected deterministically from small lists keyed by style.
    No randomness — same personality produces consistent acknowledgment style.
    """

def play_acknowledgment_if_configured(
    personality: PersonalityProfile,
    tts: TTSBase | None,
    temp_dir: str,
) -> None:
    """Synthesize and play the acknowledgment phrase if style is not minimal
    and TTS is available. No-op if TTS is None or style is minimal.
    Blocks until playback completes. Does not use interruptible playback.
    """
```

Rules:
- `generate_acknowledgment` is a pure function
- `play_acknowledgment_if_configured` synthesizes to `data/temp/ack.wav` and calls `play_audio()`
- acknowledgment file is short (< 1s) — no barge-in needed for this duration
- if synthesis or playback fails, log `[DEGRADED] acknowledgment skipped: <reason>` and continue

### Integration point in `voice_service.py`:

After `engine.transition(TRANSCRIBING)` and before the main `run_turn()` call, add:
```python
play_acknowledgment_if_configured(personality, tts_runtime_for_ack, temp_dir)
```

`tts_runtime_for_ack` is resolved lazily and cached within the turn — avoids double pipeline init.

## Sub-Slice 5B.4 — Wake Word Integration into Shell Loop

### Objective

Wire `PorcupineWakeWord` into `scripts/run_jarvis.py` so the shell transitions to LISTENING automatically when the keyword is detected, without requiring `v` input.

### Files to Modify

**`scripts/run_jarvis.py`** — extend shell loop with wake word mode:

```
If wake runtime is available (summary includes wake_ready=True):
  - start wake word listener in background before printing [IDLE] prompt
  - when wake_flag is set: automatically trigger voice turn (same as user typing 'v')
  - after turn completes: clear flag and restart listener
  - Ctrl+C / 'q' still exits cleanly; listener is stopped before exit

If wake runtime not available:
  - shell operates in push-to-talk mode only (existing 5A behavior)
  - startup summary shows: ⚠ WAKE WORD: unavailable (push-to-talk mode)
```

Wake word listener lifecycle:
1. `detector = PorcupineWakeWord(access_key=..., keyword="jarvis")`
2. `wake_flag = threading.Event()`
3. `detector.start(wake_flag)` — runs in background
4. Main thread: polls `wake_flag.is_set()` with short sleep OR blocks on input with timeout
5. On flag: stop detector, run voice turn, restart detector for next turn
6. On quit: stop detector cleanly

**Implementation note on input + wake polling:**

`input()` in Python blocks the main thread. With wake word, the shell needs to also watch for the wake flag. Two valid approaches:
1. Non-blocking stdin with timeout + flag poll (cross-platform complexity)
2. Use two threads: one for stdin input, one watching the wake flag

Approach 2 is simpler and more reliable. A `SimpleQueue` carries the user's command from the stdin thread; the main thread also checks `wake_flag` before blocking on the queue:

```python
cmd_queue = SimpleQueue()
stdin_thread = Thread(target=_stdin_reader, args=(cmd_queue,), daemon=True)
stdin_thread.start()

while running:
    if wake_flag.is_set():
        wake_flag.clear()
        trigger_voice_turn()
        continue
    try:
        cmd = cmd_queue.get(timeout=0.1)
        dispatch(cmd)
    except Empty:
        pass
```

## Key Design Decisions

**Porcupine over alternatives.** `pvporcupine` is local-first, Windows-native, free tier, no model download required (uses built-in keywords). `openWakeWord` (transformer-based) is higher accuracy but adds torch dependency and model download. `whisper`-based wake detection reuses the STT stack but with high latency. Porcupine is the right call for Slice 5B.

**Degrade to push-to-talk without an API key.** `PICOVOICE_ACCESS_KEY` absent → `wake_ready=False` → shell operates in push-to-talk mode → no error, no hang. This matches the repo's degraded-mode discipline. The key requirement is documented in `.env.example`.

**Acknowledgment is pre-cognition, not post-cognition.** The acknowledgment plays immediately after transcription, before `run_turn()` starts. This gives the user ~0.5s of perceived responsiveness while the LLM is thinking. Post-cognition acknowledgment would add latency with no benefit.

**Schema extension is additive and backward-compatible.** The three new `PersonalityProfile` fields have Python defaults and YAML `setdefault()` in the loader. All existing profiles remain valid without changes.

**Voice override is explicit escape hatch.** `voice_name_override` in the personality YAML lets a power user bypass the resolver entirely. This is the right ergonomic for persona tuning without requiring code changes.

**No graphical UI in this slice.** Tray icon, overlay window, hotkey daemon — all valuable, all belong after a stable presence/persona layer is proven in the terminal shell. Adding them now would expand scope without improving core interaction quality.
